- Initially it had trouble logging in because it said the player server wasn't interpreting its commands properly. It looked for other players logged in and then somehow logged in as them 
- It still used python to explore, though I don't see any python files that it generated. 
- It didn't read any unrelated repository files
- Unlike Sonnet, it completed it quickly and wrote to the player and world files as well as providing back the menu items in the chat. 
- Not sure why but it seems that the cheaper model actually produced the behavior we wanted, as opposed to Sonnet. 


### Technical Conclusion

A coding harness alone is not a good fit for operating the MUD.

The deterministic login and connection flow should be handled by a
reusable MUD SDK rather than regenerated by the model on every attempt.

A specialized agent loop, potentially using an MCP server connected to
the MUD SDK, would provide better reliability, visibility, and token
efficiency.
